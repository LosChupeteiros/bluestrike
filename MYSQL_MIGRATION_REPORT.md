# Feasibility Report: Supabase to MySQL Migration

Date: 2026-04-17

## Scope

This report was produced from a local analysis of the repository `e:\dev\bluestrike`, focusing on:

- current Supabase usage in the project
- functional and technical dependencies
- feasibility of migrating to MySQL
- losses in capability, architecture changes, and performance impact
- impact on request shape and application code

## Executive Summary

Yes, it is possible to migrate this project from Supabase to MySQL.

But this is not a simple provider swap. Today the project uses Supabase in three layers:

1. PostgreSQL database accessed through `@supabase/supabase-js`
2. Storage for tournament banner uploads
3. Operational convenience from the Supabase ecosystem: schema, migrations, and policies

At the same time, the project does not use some features that usually create much stronger lock-in:

- no Supabase Auth
- no Realtime
- no SQL RPC used by the app
- no client-side anon queries

That reduces migration difficulty significantly.

My conclusion:

- technical feasibility: high
- migration effort: medium to high
- risk of functional loss: low to medium
- risk of productivity and ergonomics loss: medium
- automatic performance gain: no
- impact on backend request/data-access shape: high

In short: the migration is feasible, but it should not be framed as "change the database and keep everything else the same". The frontend may barely notice, but the backend and data model will change in meaningful ways.

## Current Project Architecture

### Main stack

- Next.js 16.2.3
- React 19.2.4
- TypeScript 5
- `@supabase/supabase-js` for database and storage access
- custom JWT sessions using `jose`
- Steam OpenID authentication
- external integrations with FACEIT and Dathost

### How the project uses Supabase

Database access is centralized in an admin client:

- `src/lib/supabase/server.ts`

That client uses `SUPABASE_URL` plus `SUPABASE_SECRET_KEY` or `SUPABASE_SERVICE_ROLE_KEY`, and all access is server-side. In practice, the app relies much more on its own backend than on client-side Supabase security boundaries.

### How the project authenticates users

The project does not depend on Supabase Auth.

It uses:

- Steam OpenID for login
- its own JWT cookie in `src/lib/auth/session.ts`
- persistence of user profiles in the `profiles` table

This is very important: the login/session system is not tied to native Supabase authentication.

## PostgreSQL and Supabase Specific Dependencies

This is the most important part of the analysis.

### UUID with `gen_random_uuid()`

The schema uses PostgreSQL UUID defaults:

- `uuid primary key default gen_random_uuid()`

In MySQL this needs to become one of:

- `CHAR(36)` with UUID generated in the application
- `BINARY(16)` with binary UUIDs
- ULID or UUID generated by the backend

Loss: none functionally
Impact: low

### `text[]` arrays

The schema uses PostgreSQL arrays in:

- `profiles.faceit_team_ids`
- `tournaments.rules`
- `tournaments.tags`

MySQL has no `text[]`.

These would need to become:

- JSON columns
- or relational helper tables

Recommendation:

- `rules` and `tags`: can stay as JSON if usage remains simple
- `faceit_team_ids`: better as a relational table if this feature grows

Loss:

- native array modeling from PostgreSQL

Impact:

- medium

### `jsonb`

There is `jsonb` usage in:

- `tournaments.prize_breakdown`

In MySQL the equivalent would be `JSON`.

Loss:

- loss of advanced `jsonb` capabilities

In the current project state:

- low impact, because usage appears to be basic read/write

### `timestamptz` and `timezone('utc', now())`

The schema uses timezone-aware timestamps.

In MySQL you would standardize with:

- `DATETIME` stored in UTC
- or disciplined `TIMESTAMP` usage handled by the app

Loss:

- no major practical loss if the app enforces UTC consistently

Impact:

- low

### Triggers plus `set_updated_at()` in PL/pgSQL

The project uses triggers to maintain `updated_at`.

This exists for:

- `profiles`
- `teams`
- `tournaments`
- `matches`
- `dathost_servers`

MySQL supports triggers, but these would need to be rewritten.

A better alternative may be:

- move `updated_at` responsibility to the application or ORM

Loss:

- none functionally

Impact:

- low to medium

### `do $$ ... $$` blocks, sequences, and `setval`

The schema uses PostgreSQL-specific logic for:

- `profiles.public_id` sequence
- idempotent constraint creation

In MySQL this changes completely.

The main case is `public_id`, which can become:

- `BIGINT AUTO_INCREMENT UNIQUE`

Loss:

- none functionally

Impact:

- low

### Partial indexes and PostgreSQL-specific index semantics

There are examples such as:

- `create unique index ... where cpf is not null`
- featured-only indexes
- unread-only indexes
- `faceit_elo desc nulls last where faceit_id is not null and faceit_elo is not null`

MySQL does not offer a 1:1 equivalent for several of these patterns.

You would need to adapt with:

- composite indexes
- generated columns
- helper columns
- or accepted differences in optimizer behavior

Loss:

- some of PostgreSQL's indexing flexibility

Impact:

- medium

### RLS and policies

The schema enables row-level security and defines policies across many tables.

But looking at application code, almost everything goes through `createSupabaseAdminClient()` using admin credentials on the server.

In practice:

- the project is not strongly relying on policies as the main frontend security boundary

That means:

- losing RLS hurts less here than in apps built around anon client access

Still, it is a real loss:

- MySQL does not provide a native equivalent to PostgreSQL RLS

Authorization would become fully application-managed.

Loss:

- medium loss in declarative security capability

Impact:

- low to medium for current behavior
- high if the team wants direct client-side database access in the future

### `ILIKE` and text search

The app uses search with:

- `.ilike(...)`
- `.or(...)`
- contains filters like `%query%`

Examples:

- team search
- profile search
- tournament search

In MySQL:

- there is no `ILIKE`
- you would rely on case-insensitive collations, `LOWER(...) LIKE ...`, or FULLTEXT indexes

Important note:

- the current PostgreSQL queries are not especially optimized for massive text search either
- so moving to MySQL does not automatically make this better or worse

At larger scale, a better path would be:

- FULLTEXT in MySQL
- or a dedicated search engine later

Loss:

- low to medium

Impact:

- medium if the dataset grows significantly

## Impact on Application Code

### What would barely change

- React/Next frontend in general
- Steam authentication
- custom JWT session handling
- FACEIT and Dathost integrations
- domain types

### What would change a lot

- data-access layer
- database client initialization
- banner upload/storage flow
- parts of filtering and pagination queries
- migrations and database bootstrap

### Request shape

External HTTP contracts can remain almost unchanged.

But the internal shape of application-to-database calls changes substantially.

Today:

- `.from("table").select(...).eq(...).maybeSingle()`
- `.insert(...).select(...).single()`
- `.update(...).eq(...)`
- `.upsert(..., { onConflict: ... })`

With MySQL, the team would need a new data-access layer, such as:

- Prisma
- Drizzle
- Kysely
- `mysql2` with manual SQL

So:

- browser-to-app request shape does not need to change
- app-to-database request shape definitely changes

## Real Losses if You Move to MySQL

### 1. Loss of integrated Supabase Storage

Today upload logic uses:

- `supabase.storage.from(BUCKET).upload(...)`
- `getPublicUrl(...)`

After migration, this needs to be replaced with:

- S3
- Cloudflare R2
- MinIO
- or another object storage/CDN stack

Loss:

- operational simplicity

### 2. Loss of declarative RLS

Even though it is underused, it exists.

In MySQL:

- authorization becomes fully application code

### 3. Loss of Supabase ecosystem ergonomics

The team loses:

- integrated database plus storage dashboard
- unified environment and secret management model
- the existing Supabase client already used across the backend

### 4. Loss of portability for current migrations

The SQL migrations cannot be reused as-is.

The schema must be rewritten for MySQL 8+.

## Potential Gains if You Move

### 1. More infrastructure freedom

You stop depending on Supabase as an integrated platform and can choose:

- managed MySQL
- separate storage
- separate observability
- workload-specific tuning

### 2. Better fit if the organization already standardizes on MySQL

If the broader company stack revolves around MySQL:

- migration may simplify internal standardization

### 3. Better developer experience if you adopt a strong ORM

If the move includes:

- Prisma
- Drizzle

you may gain:

- more centralized schema definition in code
- stronger typing
- lower provider lock-in in the long run

## Performance: Better or Worse?

### Short answer

There is no evidence in this repository that moving to MySQL by itself will improve performance.

### Important point

If this system struggles at scale, the likely bottlenecks are not "Supabase vs MySQL". They are:

- query design
- index design
- `%query%` text searches
- fan-out to external APIs
- loading shape and object assembly
- repeated round-trips

### Where the project may suffer today regardless of database

- searches using `ilike '%...%'`
- loading members and profiles in multiple steps
- exact counts on listings
- repeated registration reads per tournament
- relation assembly in memory instead of using a stronger relational access pattern

These issues exist today and would still matter with MySQL.

### Where MySQL can be equivalent

With MySQL well configured and modeled correctly, you can expect similar performance for:

- basic CRUD
- pagination
- sorting on indexed columns
- upserts
- traditional joins

### Where migration can get worse

Without redesign, it may get worse in:

- flexible text search
- arrays or denormalized list fields
- cases currently helped by PostgreSQL-specific indexing

### Where migration can get better

It only tends to get better if the migration is used as an opportunity to:

- redesign data access
- revisit indexes
- revise search patterns
- reduce round-trips
- normalize some fields better

## Technical Recommendations

## Main recommendation

If the only reason is "replace Supabase with MySQL because MySQL seems better":

- I would not recommend migrating now

If the reason is:

- infrastructure standardization
- organizational requirement
- cost or control
- long-term platform strategy

then the migration is absolutely feasible.

## Lowest-risk path

### Option A: Migrate the database first, storage after

Phase 1:

- replace Supabase DB with MySQL
- keep banner storage migration as a separate step

Phase 2:

- migrate uploads and CDN carefully

### Option B: Keep PostgreSQL and leave only Supabase

If the real problem is the vendor, not PostgreSQL, this may be the best technical option.

Example:

- keep PostgreSQL
- replace Supabase with managed Postgres plus S3 or R2
- rewrite only the access layer

This option preserves better:

- arrays
- `jsonb`
- partial indexes
- current migration semantics

From a pure technical-fit perspective, this is the smoothest alternative for this repository.

## Suggested Migration Plan

### Phase 0: preparation

- choose the MySQL access stack: Prisma, Drizzle, Kysely, or manual SQL
- choose the storage provider
- freeze structural schema changes during migration

### Phase 1: model the new database

- port the active schema to MySQL
- replace arrays with JSON or helper tables
- replace `jsonb` with `JSON`
- replace `public_id` sequence with `AUTO_INCREMENT` or equivalent
- define the UUID strategy

### Phase 2: build a new data-access layer

- create adapters or repositories that hide provider details
- reimplement:
  - profiles
  - teams
  - tournaments
  - faceit_prizes
  - faceit_registrations

### Phase 3: storage

- port banner upload flow
- keep URL shape compatible where possible to minimize frontend churn

### Phase 4: backfill and dual-write if needed

- export data from PostgreSQL
- import into MySQL
- validate row counts and sample records by table
- if necessary, run a short dual-write period

### Phase 5: cutover

- switch reads to MySQL
- monitor error rate, latency, and consistency

## Effort Estimate

Conservative estimate for a careful migration, not including major side refactors:

- final analysis and target schema design: 1 to 2 days
- data-access rewrite: 3 to 5 days
- storage migration: 0.5 to 1 day
- migration and backfill scripts: 1 to 2 days
- testing, fixes, and cutover: 1 to 3 days

Plausible total range:

- 1 to 2 weeks of focused work

If the team wants lower operational risk:

- assume closer to 2 weeks

## Final Verdict

It is possible to migrate this project from Supabase to MySQL.

The project is not locked into Supabase for auth or realtime, which is excellent.

But the migration is not neutral:

- the data-access layer will be rewritten
- schema and migrations will be rewritten
- storage will be replaced
- some PostgreSQL or Supabase structural advantages will be lost

There is no evidence of automatic performance gain just from replacing Supabase with MySQL.

If the reason is strategic, the migration makes sense.

If the reason is only performance or perceived simplicity, I would first consider:

1. keeping PostgreSQL and only leaving Supabase
2. or optimizing the current queries, indexes, and storage setup first

## Objective Recommendation

- If the goal is corporate standardization on MySQL: proceed with migration
- If the goal is reducing vendor lock-in without losing current schema strengths: prefer PostgreSQL outside Supabase
- If the goal is only performance: do not migrate before addressing query patterns, indexing, and storage separately
